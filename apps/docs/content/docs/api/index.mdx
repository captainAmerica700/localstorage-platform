# Overview

The public API is intentionally small:

- `StorageManager`
- `CleanupManager`
- `StorageManagerOptions`
- `EncryptionOptions`
- `MetadataOptions`
- `Metadata`
- `StorageItem`

# Why

Most applications need one place to write, read, inspect, and clear browser
storage. The API keeps those operations explicit and namespace-aware.

# How it works

Create a `StorageManager` for reads and writes. Create a `CleanupManager` with
the same namespace when you need grouped or namespace-wide cleanup.

```ts
import { CleanupManager, StorageManager } from "localstorage-platform";

const storage = new StorageManager("commerce");
const cleanup = new CleanupManager("commerce", storage);
```

Exports:

| Export | Kind | Purpose |
| --- | --- | --- |
| `StorageManager` | Class | Writes, reads, checks, inspects, and removes values in a namespace. |
| `CleanupManager` | Class | Removes grouped records or all records owned by a namespace. |
| `StorageManagerOptions` | Interface | Configures optional value encryption for a storage manager. |
| `EncryptionOptions` | Interface | Provides the secret used by the AES encryption service. |
| `MetadataOptions` | Interface | Controls group, TTL, and encryption behavior for a write. |
| `Metadata` | Interface | Describes stored record lifecycle metadata. |
| `StorageItem<T>` | Interface | Internal persisted shape containing `value` and `meta`. |

# Usage

```ts
storage.set("cart", { lineItemIds: ["li_1", "li_2"] }, {
  group: "checkout",
  ttl: 1000 * 60 * 30,
});

const cart = storage.get<{ lineItemIds: string[] }>("cart");

cleanup.clearGroup("checkout");
```

# Best Practices

- Prefer small wrapper modules around direct manager usage.
- Treat `null` reads as normal; keys may be missing, expired, or invalid.
- Keep cleanup namespace and storage namespace identical.

# Common Mistakes

- Calling cleanup with a manager from another namespace.
- Assuming generic reads perform runtime validation.
- Using browser storage for data that must be authoritative.
